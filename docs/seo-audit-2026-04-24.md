# UtilityMega Technical SEO + Angular Performance Audit
Date: 2026-04-24
Auditor: GPT-5.3-Codex

## Scope
- Repository: `UtilityMegaAngular`
- Production site reviewed: `https://www.utilitymega.com`
- Focus: indexing, technical SEO, Angular rendering, Core Web Vitals, sitemap/crawl strategy, CTR/monetization

## Evidence snapshots
1. **Robots policy allows crawling**, includes sitemap path, and blocks `/api/` and `/*.json$`. (`public/robots.txt`)
2. **A static sitemap exists in `src/assets/sitemap.xml` but has XML syntax/schema problems**:
   - `urlset` uses sitemap schema `0.8` (outdated/non-standard for current implementations).
   - `xmlns:xsi` and `xsi:schemaLocation` are written as loose text instead of attributes on `<urlset>`.
3. **Build-time sitemap generator targets `public/sitemap.xml` with sitemap 0.9** (`scripts/generate-sitemap.js`), but `public/sitemap.xml` is absent in repo before build.
4. **All hosting configs force SPA rewrites (`/* -> /index.html`)**, which can accidentally serve HTML for `sitemap.xml` if not explicitly excluded.
5. **Live crawl check indicates route HTML is same as homepage for deep URL example** (`/calculators/emi-calculator` appeared with homepage title/content in fetched output), suggesting SSR/prerender is not consistently delivering route-specific HTML to crawlers.
6. **Canonical inconsistency risk**:
   - `index.html` canonical is `https://www.utilitymega.com`
   - `SeoService.setHomeMeta()` emits `https://utilitymega.com` (no `www`)
7. **Large metadata quality gap in content dataset** (`src/assets/data/tools.json` quick lint):
   - 115 tools total
   - 78 tool titles over 60 chars
   - 58 meta descriptions over 160 chars
   - ~41 pages with thin body copy (<120 words)

## Critical issues (High Priority)

### 1) Route rendering/indexability risk for tool pages
If Googlebot receives mostly generic shell/home content for tool URLs, indexing quality drops (soft-404/duplicate clusters, wrong title/snippet). This is the most likely cause of poor indexing.

### 2) Sitemap reliability problems
- `src/assets/sitemap.xml` is malformed and uses older schema.
- Production may not always expose real XML at `/sitemap.xml` if rewrites take precedence.
- Missing `<lastmod>` reduces crawl freshness signals.

### 3) Canonical signal inconsistency
Mixing `www` and non-`www` canonical URLs can split signals and create duplicate canonical clusters.

### 4) Meta quality debt at scale
Long titles/descriptions + thin content on many utility pages hurt relevance matching and CTR.

## Medium improvements

### 1) Angular SSR/prerender hardening
- You already have SSR scaffolding (`@angular/ssr`, `server.ts`, `main.server.ts`).
- Ensure deployment actually serves SSR output (Node runtime) OR robust prerender for all static tool routes.
- For prerender mode, generate explicit route list from `tools.json` (all category + tool URLs) so each tool gets a static HTML file.

### 2) Internal linking depth and context
- Add contextual "related calculators" blocks within article copy, not only sidebar/chips.
- Strengthen cross-category links where user intent overlaps (e.g., EMI ↔ loan comparison ↔ SIP).

### 3) Structured data coverage
- Keep `WebSite`, `Organization`, `BreadcrumbList`, `SoftwareApplication`/`WebApplication`, and `FAQPage` (only when visible FAQ exists).
- Validate JSON-LD server-side output in rendered HTML, not only post-hydration JS.

## Low priority optimizations

1. Add `hreflang` only if localized variants are launched.
2. Improve image delivery (`WebP/AVIF`, explicit width/height, preload LCP hero image).
3. Reduce JS cost for first load by stronger route-level code splitting and deferring non-critical widgets/ads.

## Step-by-step action plan

1. **Fix crawl/index foundations (Week 1)**
   - Publish exactly one valid `https://www.utilitymega.com/sitemap.xml`.
   - Ensure rewrites exclude `robots.txt`, `sitemap.xml`, and static files.
   - Keep canonical host consistently `https://www.utilitymega.com` everywhere.

2. **Guarantee indexable HTML per route (Week 1-2)**
   - Choose one strategy:
     - **SSR in production** with Node adapter and proper caching, or
     - **Full prerender** for all known tool/category pages from `tools.json`.
   - Verify output HTML for sample tool pages contains unique `<title>`, `<meta description>`, `<h1>`, main copy, and JSON-LD before JS execution.

3. **Metadata/content normalization (Week 2)**
   - Enforce title ≤ 60 chars and meta description ~140–160 chars in generator pipeline.
   - Expand thin pages to 300–600 words with use-cases, formulas, examples, FAQ.

4. **CWV + UX uplift (Week 2-3)**
   - Compress and right-size above-the-fold assets.
   - Defer heavy third-party scripts until interaction/idle where possible.
   - Stabilize layouts around ads/media slots to reduce CLS.

5. **Crawl efficiency and ranking growth (Week 3-4)**
   - Split sitemap into index + child sitemaps (categories/tools/static pages).
   - Add `lastmod` and keep updated on deploy.
   - Push new/updated URLs via IndexNow and Google Search Console resubmission.

6. **CTR + monetization alignment (ongoing)**
   - Rewrite SERP titles for benefit + intent + numbers (e.g., "EMI Calculator (2026) – Monthly Loan EMI + Amortization").
   - Add comparison tables, examples, and “download/share” hooks to increase engagement and ad viewability without CLS.

## Suggested title/meta patterns for tool pages
- **Title template**: `{{Primary Keyword}} (2026) – Free Online {{Tool Type}} | UtilityMega`
- **Meta template**: `Calculate {{intent}} instantly with our free {{tool}}. Get {{key outputs}} with formula + examples. Fast, mobile-friendly, no signup.`

## Suggested schema markup set
- Global: `Organization`, `WebSite` + `SearchAction`.
- Category pages: `CollectionPage`, `BreadcrumbList`, `ItemList`.
- Tool pages: `WebApplication` (or `SoftwareApplication`), `HowTo` (if guided steps), `FAQPage` (only visible FAQ), `BreadcrumbList`.

## Notes on environment limits during audit
- Direct terminal `curl` calls to `https://www.utilitymega.com` were blocked by a CONNECT 403 in this environment.
- Live page snapshots were therefore taken via the browsing tool.
